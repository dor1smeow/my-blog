# AGENTS.md

## Project Context

这是一个基于 Next.js App Router、Hono、Prisma 的博客项目。
协作时优先保持以下结构认知：

- 页面负责展示和页面级数据组织
- 可交互表单和按钮放在 Client Component
- 数据读取优先走现有后端链路
- API 请求优先走 Hono
- 数据落库通过 Prisma
- 非必要不要绕过现有分层直接访问数据库

## Collaboration Preferences

### 命令执行
- 常规开发命令直接执行，不要先文字询问确认。
- 包括但不限于：
  - 读取文件
  - 搜索代码
  - 查看 git diff / git status
  - 运行 lint
  - 运行 build
  - 使用本地脚本做最小验证
- 如果遇到系统级提权弹窗，直接发起命令即可，不需要先在对话里重复问我。
- 只有在以下情况才需要停下来说明风险：
  - 破坏性删除
  - git reset / 覆盖历史
  - 会覆盖我未明确要求覆盖的数据
  - 明显超出当前任务范围的大改动

### 代码修改反馈
- 每次修改代码后，按文件逐个展示改动说明。
- 不要把所有改动混成一段总结。
- 每个文件单独说明：
  - 为什么改
  - 改了什么
  - 对当前功能链路的影响
- 如果适合，可附带按文件分组的 diff 摘要。

### 沟通风格
- 先执行，再汇报。
- 少做无意义确认。
- 解释时先讲结论，再讲原因。
- 默认使用中文。
- 我是学习者，所以解释代码时：
  - 尽量分步骤
  - 尽量说清职责边界
  - 尽量说明“为什么这样设计”

### 理论优先模式
当我明确说“先解释理论再改代码”时，必须遵循：

1. 先解释相关理论
2. 再说明当前项目里这部分应该怎么落
3. 再开始改代码
4. 改完后从学习角度复盘这次实现

### 实现原则
- 优先局部修改，避免无关重构。
- 优先复用现有组件、helper、接口结构。
- 如果当前项目已有分层，不要绕过它。
- 新功能优先考虑后续复用性，但不要过度设计。
- 表单、后台管理、接口联动这类功能，优先保证主链路先闭环，再做增强。

## Frontend Preferences

- 保持界面简洁、清晰、可读。
- 后台页面优先强调信息结构和操作链路，不追求花哨视觉。
- 表单优先可维护性和清楚的状态反馈。
- 初期实现优先简单可靠，不急着引入复杂编辑器或重型依赖。

## Code Explanation Preferences

当我让你解释代码时：
- 尽量逐句解释，而不是只讲大意。
- 先解释这段代码“整体在做什么”
- 再按顺序解释每一部分
- 说清：
  - 输入是什么
  - 输出是什么
  - 中间状态是什么
  - 为什么要这样写

## Verification Preferences

- 完成改动后默认做最小必要验证。
- 优先：
  - lint
  - build
  - 最小接口验证
  - 关键链路验证
- 如果没法验证，要明确告诉我缺了什么条件。
- 如果验证失败，不要只汇报失败，要继续定位并尽量修到可用。

## Context Discipline

- 保持工作上下文精简，优先只读取当前任务必需的信息。
- 读取代码时优先使用搜索、定点读取、局部展开，不要无差别加载整批文件。
- 面对大文件时先读取相关片段，确认需要后再继续向下展开。
- 在继续读取更多内容前，先复用和总结已经拿到的上下文，避免重复读取。
- 除非任务明确需要，否则避免加载大型生成文件、长日志和无关目录。
- 代码解释和改动说明优先压缩为高信号摘要，不要把低相关背景一并带入后续上下文。

## Admin Feature Guidance

实现后台功能时，优先按这个顺序思考：

1. 页面展示
2. 交互组件
3. API 调用
4. Hono 路由
5. Prisma / 数据层
6. 操作成功后的刷新或跳转

后台功能默认要考虑：
- 加载状态
- 成功提示
- 失败提示
- 禁用态
- 页面刷新或跳转闭环

## Form Guidance

创建和编辑类功能优先遵循：

- 优先抽共享表单组件
- 使用清晰的 props，如：
  - `mode`
  - `initialValues`
  - `categories`
  - `tags`
- 先保证字段完整、提交流程打通
- 初期优先用基础输入控件
- 不要过早引入富文本编辑器
